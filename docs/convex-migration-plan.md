# Convex Migration Plan

This file records how the project moved from a local JSON prototype to a live Convex multiplayer backend. For the fastest current handoff, read `docs/project-handoff.md` first.

The migration is now mostly complete. The current app is Convex plus a static GitHub Pages frontend.

## Why Convex

Convex gives us three things the game needs in one place:

- A shared database for all players.
- Server functions for trusted game actions like training units and launching raids.
- Scheduled functions for delayed events like raids landing and Plateau Runs resolving.

## Retired Local Pieces

The old local JSON server, local save file, and one-off local server tests have been removed from the active project tree.

That earlier scaffold was useful for learning and prototyping, but the current deployment path is now Convex plus a static frontend.

## Current Convex Pieces

- `convex/schema.ts`: tables for players, units, buildings, raids, messages, and Plateau Runs.
- `convex/game.ts`: queries for reading the player/world state.
- `convex/players.ts`: account/profile setup and player actions.
- `convex/army.ts`: training and unit management.
- `convex/raids.ts`: launch raids and resolve raids.
- `convex/plateauRuns.ts`: create, join, and resolve Plateau Runs.
- `convex/messages.ts`: inbox and notifications.
- `convex/crons.ts`: recurring scheduled jobs.
- `convex/admin.ts`: single-admin email gate.

## Auth Migration

The project is moving from prototype name-based login to real account ownership.

- Convex Auth scaffold added:
  - `convex/auth.ts`
  - `convex/http.ts`
  - `authTables` in `convex/schema.ts`
- `players.authUserId` is the bridge between an authenticated account and one warcamp.
- `players.authUserId` should use Convex Auth's stable `getAuthUserId(ctx)` value, not the raw token identifier.
- Authenticated functions now use clean names and derive the acting player from the signed-in account:
  - `players.createPlayer`
  - `players.getPlayer`
  - `players.getDashboard`
  - `army.trainUnit`
  - `buildings.upgradeBuilding`
  - `raids.launchOpenAcreRaid`
  - `raids.launchSphereRaid`
  - `raids.launchPlayerRaid`
  - `raids.listVisibleRaids`
  - `messages.listInbox`
  - `messages.sendMessage`
  - `messages.markInboxRead`
  - `plateauRuns.joinPlateauRun`
  - `economy.settlePlayer`
- The old browser-supplied `playerId` action paths have been removed from player-owned game actions.
- Convex Auth has been installed locally and generated by `npx convex dev`.
- Browser dashboard now signs in through Convex Auth and calls the clean auth-owned function names.
- Later step: add more admin/dev-only maintenance tools such as world reset and old-data cleanup, with explicit access control.

## Admin Tools

For now, the project uses one simple admin gate instead of full player roles.

- Admin access is controlled by the Convex environment variable `ADMIN_EMAILS`.
- Multiple admin emails can be comma-separated.
- The browser hides the Testing tab from non-admins.
- The server still protects the testing functions directly, so hiding the button is not the only safeguard.
- Current admin-protected helpers:
  - `raids.forceResolveRaid`
  - `plateauRuns.startPlateauRun`
  - `plateauRuns.forceResolvePlateauRun`
  - `economy.advanceEconomy`

To choose the admin account, use the same email that signs in through Convex Auth:

```powershell
npx convex env set ADMIN_EMAILS "your-email@example.com"
```

Later, this can grow into a `roles` field or admin table with roles like admin, moderator, and player.

## Time And Economy Model

Use a hybrid lazy/scheduled model:

- The game clock is derived from the shared world's `createdAt` time.
- Ordinary sphere income should use lazy settlement, not constant background writes.
- Player actions that spend or depend on spheres should settle that player's economy first.
- `army.trainUnit` and `buildings.upgradeBuilding` now settle player economy before checking costs.
- `economy.settlePlayer` can explicitly collect pending income for one player.
- Existing test players without `lastEconomyAt` fall back to their `createdAt` timestamp the first time economy is checked.
- Raids should use scheduled functions because they are specific future events.
- Plateau Runs use scheduled functions for open, close, and resolve events.
- Scheduled Plateau Runs are checked every five minutes and open during the noon and 8 PM Mountain time windows.
- Later hourly-style ticks can be added for systems that benefit from a visible rhythm, but they should not be the default way to update every player.

The goal is to keep the backend clean while preserving the feeling of a living real-time game.

## Teaching Notes

Convex changes the mental model:

- Instead of editing a JSON file, we write documents into tables.
- Instead of browser buttons calling our local Node routes, browser buttons call Convex functions.
- Instead of manually checking whether raids have arrived, Convex can schedule a future resolver when the raid is launched.

The key rule for multiplayer: the browser can ask for an action, but trusted game logic should happen on the server side.
